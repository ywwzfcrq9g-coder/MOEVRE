Processanalys

En reflektion och analys av kodningsarbetet bakom denna webbplats, med fokus pa de val jag gjort inom HTML5 och CSS.

Struktur och semantisk HTML

Ett av de forsta besluten jag tog var att anvanda semantiska HTML5-element genomgaende pa alla sidor. Istallet for att bygga hela layouten med generiska <div>-element valde jag element som <header>, <nav>, <main>, <article> och <footer>. Anledningen ar att semantisk HTML forbattrar tillgangligheten markant — skarmlasare som NVDA och VoiceOver kan navigera sidan mer effektivt nar strukturen ar tydlig och meningsfull (MDN Web Docs, 2023). Utan semantiska element ser skärmläsare bara en generisk behållare och kan inte skilja navigering från innehåll, vilket försvårar orientering på sidan.

For navigeringsmenyn valde jag en oordnad lista (<ul>) med <li>-element inuti ett <nav>-element. Detta ar den vedertagna standarden for menyer pa webben. En meny ar en lista av alternativ utan inbordes rangordning, och <ul> kommunicerar detta tydligt till bade webblasare och hjalpmedel. Alternativet hade varit att anvanda <a>-taggar direkt utan liststruktur, men det hade forsamrat tillgangligheten eftersom skärmläsare då inte kan meddela hur många alternativ som finns i navigeringen (W3C, 2023). Jag lade ocksa till aria-label="Huvudmeny" pa <nav>-elementet, vilket ytterligare tydliggör syftet for hjälpmedel.

För processanalysen använde jag <article>-elementet, vilket är semantiskt korrekt för ett självständigt textstycke som kan stå för sig självt. För bilder använde jag alltid alt-attribut med beskrivande text, vilket är ett krav enligt WCAG 2.1 för att göra innehåll tillgängligt för synskadade användare (W3C, 2023).

En primar CSS-fil och CSS-variabler

All CSS samlas i en primär fil, styles.css, som länkas in i varje HTML-fil via ett <link rel="stylesheet">-element i <head>. Detta är effektivare än att upprepa stilar i varje fil på två sätt. För det första cachar webbläsaren CSS-filen efter första laddningen — alla efterföljande sidor laddas då utan att behöva hämta om stilarna från servern, vilket ger snabbare sidladdning (Duckett, 2011). För det andra innebär en gemensam fil att en ändring — till exempel av färgen på primärfärgen — automatiskt slår igenom på alla sidor utan att man behöver redigera varje HTML-fil separat.

Jag använde CSS custom properties (variabler) definierade i :root för att samla återkommande värden på ett ställe. Variabler som --primary, --accent, --bg och --radius används genomgående i hela stilmallen. Denna princip kallas DRY — Don't Repeat Yourself — och är fundamental i modern webbutveckling (Duckett, 2011). Om man senare vill byta ut exempelvis primärfärgen räcker det att ändra ett enda värde i :root, istället för att söka och ersätta samma färgkod på hundratals ställen. Det minskar risken för misstag och gör koden enklare att underhålla.

Responsiv design och media queries

Webbplatsen är responsiv och anpassar sig till tre skärmstorlekar: desktop (över 860px), tablet (640–860px) och mobil (under 640px). Jag valde dessutom en fjärde brytpunkt vid 400px för mycket små mobiler. Brytpunkterna valdes baserat på vanliga enhetsstorlekar snarare än specifika enhetsmodeller, vilket är den rekommenderade approachen — man designar för innehållet, inte för en specifik iPhone-modell (MDN Web Docs, 2023). Fördelen med detta är att layouten fungerar även på framtida enheter med skärmstorlekar som ännu inte existerar.

På mobil ersätts den horisontella navigeringen med en hamburger-meny som öppnas via JavaScript. Knappen är implementerad som ett semantiskt <button>-element med aria-expanded och aria-controls-attribut, vilket kommunicerar menyns status till skärmläsare. Menyns synlighet styrs med klassen .open som läggs till och tas bort via classList.toggle() — ett minimalt och effektivt JavaScript-mönster utan behov av externa bibliotek.

Jag använde clamp() för fontstorlekar, till exempel font-size: clamp(36px, 6vw, 64px) för huvudrubriken. Funktionen definierar ett minimum, ett idealt värde och ett maximum — texten skalas flytande med skärmbredden utan att man behöver skriva separata media queries för varje textstorlek. Detta kallas flytande typografi och ger en mer organisk upplevelse än skarpa hopp vid fasta brytpunkter (MDN Web Docs, 2023). Ingen fast höjd är satt på innehållscontainers — höjden anpassar sig automatiskt efter mängden innehåll.

Typsnitt och Google Fonts

Jag valde Inter från Google Fonts som huvudtypsnitt för hela webbplatsen. Inter är designat specifikt för skärmanvändning och är optimerat för läsbarhet på olika skärmstorlekar och upplösningar (Andersson, 2016). Det är ett geometriskt sans-serif typsnitt med hög x-höjd, vilket gör det lättläst även i små storlekar på mobilskärmar. Alternativa typsnitt jag övervägde var Roboto och Source Sans, men Inter valdes för att det upplevdes som mer neutralt och modernt i kombination med webbplatsens rena designstil.

Typsnittet laddas in effektivt med rel="preconnect" för att förbereda anslutningen till Googles servrar redan under sidladdningens tidiga fas. Som fallback i font-family-deklarationen anges ui-sans-serif, system-ui, sans-serif — sidan förblir läsbar även utan internetanslutning till externa resurser, eftersom webbläsaren då faller tillbaka på operativsystemets egna typsnitt.

Radlängden i processanalysen är begränsad till 720px via max-width på .process-article. Bringhurst (2004) skriver att den optimala radlängden för löptext är 45–75 tecken per rad — en för lång rad gör det svårt för ögat att hitta tillbaka till nästa rad, medan en för kort rad skapar ryckiga ögonrörelser. Kombinationen av begränsad bredd och line-height: 1.8 ger ett luftigt och behagligt läsflöde.

Layout med Flexbox och CSS Grid

Jag använde Flexbox och CSS Grid som de två huvudsakliga layoutverktygen, valda utifrån principen att använda rätt verktyg för rätt uppgift (MDN Web Docs, 2023). Flexbox används för endimensionella layouter — det vill säga layouter längs en axel i taget. Navigeringslänkarna, projektkortens text-bild-kombination och footerns märkeskolumn är alla Flexbox-layouter. Fördelen med Flexbox är att elementen automatiskt justerar sin storlek och position i förhållande till varandra, vilket gör det enkelt att centrera och fördela utrymme.

CSS Grid används för tvådimensionella layouter — layouter som spänner över både rader och kolumner. Footern är uppdelad i ett tre-kolumners Grid på desktop, som komprimeras till två kolumner på tablet och en kolumn på mobil. Grid ger mer kontroll än Flexbox när man behöver styra placering i båda dimensionerna samtidigt (MDN Web Docs, 2023). Projektmetadatan på projektsidorna är också ett Grid med två kolumner, vilket ger ett strukturerat och lättläst utseende.

För projektbilderna användes object-fit: cover som fyller bildytan utan att förvränga bilden oavsett dess ursprungliga proportioner. Utan object-fit: cover riskerar bilder att sträckas eller klämmas ihop när de placeras i behållare med fasta dimensioner. Egenskapen flex-shrink: 0 på bilderna säkerställer att de inte krymper när utrymmet är begränsat, utan att layouten istället staplas vertikalt via media queries.

Utskriftsstod via @media print

Via @media print döljer jag element som inte är relevanta vid utskrift: headern med navigeringen, hero-bakgrundsbilden och projektbilderna i korten. Navigeringen är meningslös på papper eftersom man inte kan klicka på länkar. Stora bakgrundsbilder är onödiga och kostsamma att skriva ut — de förbrukar mycket bläck utan att tillföra informationsvärde. Kvar vid utskrift finns rubriker, brödtext och kontaktuppgifter i footern.

Bakgrundsfärgen ändras till vit (#fff) vid utskrift med !important för att säkerställa att den slår igenom även om webbläsaren har egna inställningar. Padding-top nollställs på body så att topp-marginalen från den fixerade headern inte skapar ett tomt utrymme överst på utskriften. Att tänka på utskrift är en del av god webbstandard — många användare skriver fortfarande ut webbsidor, till exempel för att spara kontaktinformation (Duckett, 2011).

Box-sizing och CSS reset

I början av styles.css sätter jag box-sizing: border-box på alla element via universalväljaren *. Standardbeteendet i CSS är content-box, vilket innebär att padding och border läggs till utanför elementets angivna bredd — ett element med width: 100% och padding: 20px blir då 140px bred istället för 100px. Med border-box räknas padding IN i bredden, vilket gör layoutberäkningar mer förutsägbara och intuitive (Duckett, 2011). Detta är idag industristandardpraxis och rekommenderas av MDN Web Docs.

Jag nollställer margin och padding på alla element för att ta bort webbläsarnas inbyggda standardstilar. Olika webbläsare har olika standardvärden för dessa egenskaper, vilket kan leda till inkonsekvent utseende. En reset ger en neutral bas att bygga vidare på. Jag valde en minimal reset — bara box-sizing, margin och padding — snarare än en fullständig reset-fil som normalize.css, eftersom den minimala varianten är lättare att förstå och ger tillräcklig kontroll för detta projekt.

Reflektion

Det jag lart mig mest av detta projekt ar vikten av att ha all CSS i en gemensam extern fil. Under arbetet provade jag inledningsvis att skriva CSS direkt inline i varje HTML-fil, men insåg snabbt att detta innebar att samma kod kopierades till varje sida — varje ändring behövde göras på sju ställen istället för ett. Övergången till en gemensam styles.css löste detta och illustrerade DRY-principen i praktiken på ett mycket konkret sätt.

Att kommentera koden visade sig vara viktigare än jag förväntade mig. När jag återvände till delar av koden efter ett par dagars uppehåll var kommentarerna avgörande för att snabbt förstå varför ett visst val gjordes — till exempel varför overflow-x: clip används istället för overflow-x: hidden (clip bryter inte position: sticky). Utan kommentarer hade det tagit lång tid att lista ut sådana detaljer igen.

Projektet har gett mig en djupare förståelse för hur HTML5 och CSS samarbetar för att skapa välstrukturerade, tillgängliga och responsiva webbplatser. Fokus på semantik, konsistens och underhållbar kod är principer jag tar med mig i framtida projekt. Jag har också fått en starkare känsla för när man bör välja Flexbox respektive Grid, och hur media queries bäst skrivs utifrån innehållets behov snarare än specifika enhetsstorlekar.
